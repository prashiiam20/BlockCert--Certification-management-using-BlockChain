ABSTRACT

The issue of academic credential forgery remains a significant vulnerability in the global educational and recruitment ecosystems. Traditional methods of verifying educational certificates are manual, time-consuming, and prone to centralized hacking or institutional database failures. To address this, we present "BlockCert," a decentralized academic credential registry built securely on the Ethereum Blockchain and the InterPlanetary File System (IPFS).

BlockCert represents an end-to-end Web3 platform designed to eliminate degree forgery. The system allows verified educational institutions to hash and encrypt student academic records client-side using a highly performant ChaCha20-Poly1305 AEAD algorithm. The encrypted records are perpetually anchored to IPFS, while the resulting hashes and metadata are committed to Ethereum Smart Contracts as an immutable proof-of-authenticity. Employing an Institutional Registry Vault compatible with EIP-191 signatures, BlockCert shifts key-management sovereignty fully to the issuer.

For employers and recruiters, the platform provides a completely frictionless 'Deep-Linked QR Verification' pipeline. By simply scanning a dynamic Quick Response token directly from the student's degree, the system computationally fetches the respective IPFS payload, applies the decryption matrix using localized sharding, and validates the credential against the on-chain registry in under a second. BlockCert thereby bridges advanced Zero-Knowledge principles with Web3 role-based access control to formulate an impenetrable, accessible, and instantaneous verification solution.

1. INTRODUCTION

1.1 About Project

The rapid digitization of academic infrastructures has simultaneously increased the prevalence of fake degrees and altered credentials. Traditional paper-based certificates can be easily falsified utilizing modern graphic design tools, while centralized academic databases present a single point of failure and are highly susceptible to malicious breaches or unauthorized data alteration. BlockCert introduces a comprehensive Web3-native solution to this problem by utilizing the decentralized, immutable, and cryptographically secure properties of the Ethereum Blockchain.

BlockCert is a Decentralized Academic Credential Registry system that securely anchors digital academic credentials on a distributed ledger. Using Ethereum Smart Contracts to handle the business logic and InterPlanetary File System (IPFS) to manage scalable, off-chain, decentralized storage, the platform guarantees that a certificate, once issued, can never be altered or retroactively tampered with, without breaking the underlying cryptographic hashes.

1.2 Scope and Objective

Scope: The scope of this project encompasses the creation of an interactive digital registry for Institutions, Students, and Recruiters. The scope is limited to verified issuance by granted institutional wallets, real-time client-side encryption of the documents, uploading via Pinata to IPFS, and a fast-verification scanning mechanism primarily intended for automated recruitment background checks.

Objectives:

  • To design an immutable structure for certificate issuance using Solidity smart contracts.
  • To implement Zero-Knowledge Cryptographic Middleware (ChaCha20-Poly1305) to ensure student privacy.
  • To integrate IPFS for highly available, decentralized file storage.
  • To construct a frictionless Deep-Linked QR Verification system, allowing verification in under a second.
  • To enforce strict Identity and Role-Based Access Control (RBAC) securely on-chain.

2. LITERATURE SURVEY

The concept of utilizing blockchain technology for academic tracking is not entirely novel, though implementations have vastly differed in privacy, scalability, and centralization. Earlier implementations often heavily relied on traditional centralized databases bridged loosely to a private blockchain. This limits transparency and creates localized trust dependencies.

One of the earlier proposals is MIT's "Blockcerts" standard (Open Badges). While functionally robust, these earlier iterations frequently expose plaintext JSON payloads on a public ledger, which structurally violates data privacy laws like GDPR because the blockchain is immutable. Other academic setups attempt to store large amounts of metadata directly on-chain, leading to massive gas fees and poor network scalability.

BlockCert innovates upon these previous works by utilizing severe client-side encryption via `@noble/ciphers` (ChaCha20-Poly1305) before any data touches IPFS. While previous works rely on centralized databases for querying metadata, BlockCert strictly enforces off-chain payload storage connected strictly by on-chain Merkle Roots and deterministic Keccak256 hashes.

Furthermore, standard identity models in existing literature rely heavily on centralized PKI (Public Key Infrastructure). Through an EIP-191 derived "Institutional Registry Vault", BlockCert permits institutions to mathematically derive cryptographic sharding using solely their Web3 wallet signature, circumventing centralized database hacks completely.

The System Design outlines the architectural blueprints of the BlockCert DApp. It involves defining how different modules like the User Interface, IPFS Gateway, and Smart Contracts communicate synchronously. Modeling is done primarily through standard UML diagrams, highlighting actor-to-system boundary crossing.

4.1 UML Diagrams

Unified Modeling Language (UML) is a standardized modeling language enabling developers to specify, visualize, construct, and document the artifacts of a software system. The following nine diagrams define the system from multiple perspectives.

1. Use Case Diagram

The Use Case Diagram defines the primary actors: Admin, Institution, Student, and Recruiter. An Admin oversees the system and distributes verification roles. An Institution issues the cryptographic credentials and invokes revocation if required. A Student views their academic records, while the Recruiter utilizes the QR token to decrypt and verify the document without needing explicit access keys.

2. Class Diagram

Outlines the object-oriented structure of the Smart Contract system. It highlights the `Certificate` struct encapsulating the `ipfsCID`, `certHash`, `student` address, and `merkleRoot`. The `CertificateRegistry` class maps `bytes32 -> Certificate` and enforces access control inheritance from the `AccessControl` utility contract.

3. Sequence Diagram

Visualizes the linear progression of Certificate issuance. The Institution dispatches the raw file, the middleware compresses (ZLib) and encrypts it (ChaCha20), pushes to Pinata (IPFS), returns the CID, and sequentially dispatches a decentralized transaction via Metamask to the `CertificateRegistry` contract on the Ethereum network.

4. Activity Diagram

Details the conditional flow of verification. The Recruiter inputs a hash. The system queries the Ethereum blockchain. If the hash exists, it checks if it is revoked. If valid, it pulls the encrypted IPFS CID string, decrypts it client-side into a PDF/Image, and renders a "Valid Green" verification token on the screen.

5. State Chart Diagram

Shows the state transistion of a Certificate object within the mapped storage. A certificate moves from an uninitialized "Null" state, to "Issued" (timestamped), and potentially to "Revoked" (if boolean revoked triggers true). Furthermore, an expiry state can be reached based on block timestamps.

6. Component Diagram

Demonstrates the distinct micro-architecture wiring. Shows the React Frontend communicating over JSON-RPC via Web3/Ethers.js to the Hardhat Ethereum Node, while concurrently making RESTful requests to the Pinata IPFS gateway for file anchoring.

7. Deployment Diagram

Visualizes the hardware infrastructure. Client Browsers connect via internet to the Vercel CDN hosting the React App. The React App interfaces directly with the Mainnet/Testnet EVM via providers like Infura or Alchemy, keeping the infrastructure entirely decentralized.

8. Collaboration Diagram

Shows the spatial interaction of the network, emphasizing the message passing between the Admin wallet granting the `Role.INSTITUTION` to a verified address before the address can successfully sign a state-changing transaction on the EVM.

9. Data Flow Diagram (DFD)

Represents the flow of data across system levels. Level 0 depicts the entire platform as a single process. Level 1 breaks down the data flow between Encryption engines, Storage networks, Transaction mempools, and User interfaces.

5. SYSTEM IMPLEMENTATION

Implementation executes the logical model into functional Solidity and Javascript code. The core blockchain runtime was programmed using Solidity `^0.8.20`, implementing modern optimization and Merkle tree mechanisms. The frontend was built with React.js using standard Hooks for asynchronous wallet handling.

5.1 Sample Code

The core `CertificateRegistry.sol` contract outlining the structural parameters of cryptographic credentials: 

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "./AccessControl.sol";
import "@openzeppelin/contracts/utils/cryptography/MerkleProof.sol";

contract CertificateRegistry is AccessControl {
    struct Certificate {
        string ipfsCID;
        bytes32 certHash;
        address student;
        address institution;
        uint256 issueDate;
        uint256 expiryDate;
        bool revoked;
        bytes32 merkleRoot;
    }
    
    mapping(bytes32 => Certificate) public certificates;
    mapping(address => bytes32[]) public studentCertificates;
    
    event CertificateIssued(bytes32 indexed certId, address student, address institution);
    
    function issueCertificate(
        string memory ipfsCID,
        bytes32 certHash,
        address student,
        uint256 expiryDate
    ) external onlyRole(Role.INSTITUTION) returns (bytes32) {
        bytes32 certId = keccak256(abi.encodePacked(certHash, student, block.timestamp));
        
        certificates[certId] = Certificate({
            ipfsCID: ipfsCID,
            certHash: certHash,
            student: student,
            institution: msg.sender,
            issueDate: block.timestamp,
            expiryDate: expiryDate,
            revoked: false,
            merkleRoot: bytes32(0)
        });
        
        studentCertificates[student].push(certId);
        emit CertificateIssued(certId, student, msg.sender);
        
        return certId;
    }
}

7. RESULTS

The completed BlockCert ecosystem demonstrates significant advantages over both legacy database models and primitive on-chain storage methodologies. After successful compilation and test-net deployment, the smart contracts showcase an extremely optimal state size and low algorithmic gas costs, allowing educational bodies to mass-deploy credential proofs economically.

The frontend application achieves strict sub-second encryption targets via Zero-Knowledge architectures utilizing ChaCha20-Poly1305. Uploads to Pinata's InterPlanetary File System consistently propagate globally. Crucially, the deep-linked QR resolution permits end-users with no crypto-knowledge to verify academic degrees natively from any device. When tested, verified certificates display as "Green," immediately returning the decoded, verifiable document to recruiters, completely eliminating manual third-party cross-checks.

8. CONCLUSION & FUTURE SCOPE

Conclusion: BlockCert resolves the pervasive challenge of credential forgery by utilizing the inherent properties of decentralized consensus networks. By binding cryptographic document anchors, Zero-Knowledge file encryption, and structural Roll-Based Access Control smart contracts into a streamlined user application, the system abstracts the complications of Web3 away from institutions while providing them with unparalleled registry security. The platform entirely avoids localized database breaches and empowers organizations to mint and maintain their academic sovereignty.

Future Scope: The project possesses a robust foundation capable of vast expansions. Key avenues for future scope include:

  • ZK-Rollups and Layer 2s: Migrating the underlying logic to an Ethereum L2 (such as Arbitrum or Polygon) to reduce gas issuance fees to fractions of a cent, allowing thousands of batch mints.
  • Dynamic Soulbound Tokens (SBTs): Rather than just standard verification registries, future versions could mint non-transferable Soulbound NFTs directly to the graduate's personal digital wallet, establishing deep professional identities.
  • Institutional Consortiums: Extending the Access Control smart contracts to incorporate decentralized voting by governing bodies to admit new universities automatically via DAO-style consensus protocols.
